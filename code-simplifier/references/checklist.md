# 代码精简清单 · 详细判定标准

> 每条都给出"信号 + 怎么改 + 不要改的例外"。

---

## P0 · 几乎一定要做

### P0.1 · 未使用的导出
- **信号**:跨文件 grep 该 symbol,只在自身文件出现
- **改法**:删除 export 关键字,或彻底删除函数
- **例外**:库代码暴露的公共 API(看 `package.json` exports / `index.ts`)

### P0.2 · 未使用的导入
- **信号**:导入了但文件里没用
- **改法**:删除 import 语句
- **例外**:有副作用的 import(如 `import './polyfill'`)

### P0.3 · 未使用的参数 / 局部变量
- **信号**:linter 警告,或人工 grep 发现
- **改法**:
  - 函数参数 → 删除(若签名不能改,前缀 `_`)
  - 局部变量 → 删除赋值
- **例外**:接口契约要求的参数(如 React event handler)

### P0.4 · 死代码分支
- **信号**:`if (false)`、永远不进入的 case、永远 throw 的兜底
- **改法**:删除分支
- **例外**:故意保留作为占位(必须有注释解释)

### P0.5 · 死函数
- **信号**:全仓库无调用点,且不是导出 API
- **改法**:删除整个函数
- **例外**:测试 helper、CLI entry point

### P0.6 · 重复代码块(3 次以上)
- **信号**:3 段或以上**结构相同**的代码,差异只在数据
- **改法**:抽出函数,注意命名要描述"做什么"而非"怎么做"
- **例外**:看似重复但语义不同(巧合的相似)

### P0.7 · 兼容垫片(已无调用方)
- **信号**:`legacy_xxx`、`old_xxx`、`v1_xxx` 这种命名 + 注释里说"为了兼容"
- **改法**:确认无调用方后删除
- **例外**:暴露给外部用户的 API(需要 deprecation 周期)

### P0.8 · 引用已不存在代码的注释
- **信号**:`// see foo()` 但 foo 已删除,`TODO(#1234)` 但 issue 已关
- **改法**:删除注释,或更新到当前现实

## P1 · 大概率要做

### P1.1 · 单实现 interface
- **信号**:`interface Foo` 只有一个 `class FooImpl`
- **改法**:删除 interface,直接用 class
- **例外**:为测试预留的 mock 边界(测试里有 mock 实现)

### P1.2 · 单调用点 helper
- **信号**:只在一处被调用的小函数
- **改法**:内联到调用点(若内联后可读性下降则保留)
- **例外**:函数名本身就是好文档(命名替代了一段注释)

### P1.3 · 过度防御
- **信号**:在 private/internal 函数里检查 null、类型、范围,但调用方已保证
- **改法**:删除检查
- **例外**:边界(用户输入、外部 API、跨进程)的检查必须保留

### P1.4 · 空壳包装
- **信号**:函数体只有一行 `return foo(args)` 且没改语义
- **改法**:删除包装,直接调 foo
- **例外**:为了 API 稳定性而保留(显式 stable 注释)

### P1.5 · 墓碑注释
- **信号**:`// removed xxx`、`// was: foo`、`// kept for backward compat`(但已无 caller)
- **改法**:连注释带代码一起删
- **例外**:有 issue 编号且 issue 还活着

### P1.6 · 重复的常量定义
- **信号**:多个文件定义同名常量,值一样
- **改法**:抽到 constants 文件,统一 import
- **例外**:语义实际不同,只是巧合相同

### P1.7 · 冗余的类型断言
- **信号**:`x as Foo` 但 x 已经是 Foo 类型
- **改法**:删除 `as`
- **例外**:跨包边界、类型推断不通过的场景

## P2 · 谨慎判断

### P2.1 · 嵌套 if 改卫语句
- **信号**:3 层及以上的嵌套 if
- **改法**:提前 return
- **例外**:嵌套确实更清晰(罕见)

### P2.2 · 复杂布尔条件
- **信号**:`!a || (b && !c) || ...` 这种
- **改法**:抽变量、用德摩根、拆函数
- **例外**:已经是最简形式

### P2.3 · 循环改函数式
- **信号**:for 循环 + 累加器,可改 reduce/map/filter
- **改法**:函数式版本
- **例外**:函数式版本更难读(性能敏感场景也保留 for)

### P2.4 · 拆函数
- **信号**:函数超过 80 行 + 内部有明显的"段落"
- **改法**:每段抽一个函数
- **例外**:抽出的函数没有好名字,或参数列表过长

### P2.5 · 合并函数
- **信号**:两个函数总是被一起调用,且没人单独调过其中一个
- **改法**:合并
- **例外**:两个函数有独立的命名价值

## 不要做(看似简化但实非)

| 操作 | 为什么不做 |
|------|-----------|
| if/else → 三元 | 可读性下降 |
| 多行变量 → 一行 | 可读性下降 |
| 删边界检查 | 那是边界,不是过度 |
| 全仓库改成箭头函数 | 风格差异不是简化 |
| 函数参数全改解构 | 可能破坏调用方 |
| 把 const 全部改成 let / var | 反向操作,且可能改变行为 |
| 删除"看起来没用"的依赖 | 可能有间接使用,要 grep 确认 |
| 重排导入顺序 | 这是格式化的事,不是简化 |

---

## 决策流程

每条候选改动跑这个流程:

```
1. 这个改动属于哪个 P 级别?(P0/P1/P2)
2. 风险:有没有调用方/测试会踩到?
3. 改动后能跑哪个验证确认无 regression?
4. 如果保留,理由能在一句话内说清吗?
   ├─ 能说清 → 保留,记入"保留未改"列表
   └─ 说不清 → 改
5. 改完跑验证,失败就回滚这一组
```
